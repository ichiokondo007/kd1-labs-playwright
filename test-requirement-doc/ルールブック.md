# Playwright セレクタ利用ガイド

このガイドは、E2E テストの安定性・保守性を高めるために、Playwright におけるセレクタの使い方を標準化することを目的としています。Playwright 公式ドキュメントの推奨事項に基づき、**Role → TestId → Text → CSS** の優先順位でセレクタを利用します。

## 標準化目的

1. フレーキー対策
2. 保守維持管理観点
3. できる限り、開発者以外でも確認できる、テスト設計書として利用を視野。

---

## 🎯 セレクタ利用の基本方針

Playwright では、次の優先順位でセレクタを選択します。

1. **Role セレクタ（最優先）**
2. **data-testid（固定IDのみ）**
3. **テキストベース（hasText / name）**
4. **CSS / class（最終手段）**

この順番は、UI 変更に強く、テストのフレイキーを防ぐためのベストプラクティスです。

---

## 🚀 Role セレクタ（最優先）

### 使い方

```ts
page.getByRole('button', { name: '保存' })
page.getByRole('textbox', { name: 'ユーザー名' })
page.getByRole('heading', { name: 'ユーザー一覧' })
```

### 利点

- UI 変更に強い  
- アクセシビリティ対応と相性が良い  
- Playwright が最も最適化しているセレクタ

### 適用対象

- ボタン  
- 入力欄  
- 見出し  
- タブ  
- モーダル

---

## 🚀 data-testid（固定IDのみ）

### 使い方

```ts
page.getByTestId('page-user-list')
page.getByTestId('btn-save')
page.getByTestId('modal-confirm')
```

### 利点

- テスト専用の安定した識別子  
- UI 変更に強い  
- 命名規則を統一しやすい

### 注意点

- **動的IDを入れない（絶対に）**  
- 一覧の行など動的要素には使わない

### 推奨命名規則

- 画面ID：`page-xxx`  
- ボタン：`btn-xxx`  
- モーダル：`modal-xxx`  
- タブ：`tab-xxx`

---

## 🚀 テキストベース（hasText / name）

### 使い方

```ts
page.locator('tr', { hasText: '12345' }).click();
page.getByRole('row', { name: /12345/ }).click();
```

### 利点

- 一覧の行など動的要素に最適  
- レコードIDや名前で行を特定できる

### 適用対象

- テーブル行  
- リスト  
- カード  
- 動的に増減する要素

---

## 🚀 CSS / class（最終手段）

### 使い方

```ts
page.locator('.user-card .title')
page.locator('#main-content .btn-primary')
```

### デメリット

- UI 変更に弱い  
- class 名が変わるとテストが壊れる  
- BEM など命名規則が厳密でないと危険

---

## 🚀 一覧画面での推奨パターン

### 行全体をクリック

```ts
page.locator('tr', { hasText: '12345' }).click();
```

### 行内のボタンをクリック

```ts
const row = page.locator('tr', { hasText: '12345' });
row.getByRole('button', { name: '編集' }).click();
```

# 🚀 実践的なセレクタ利用例（重要）

## 👁️ 利用者の目線でテストを書く（Playwright の思想）

Playwright の公式は「ユーザーが実際に見る情報を使ってテストを書く」ことを強く推奨しています。  
そのため、**親要素 → 子要素** の順で自然に UI を辿る書き方は非常に有効。

---

## 🚀 親要素で絞り込んでから getByRole を使う

### 例：

1. 「基本情報」という名前のセクションを見つける
2. その中にある「ログイン」ボタンをクリックする

```ts
await page
  .getByRole('group', { name: '基本情報' }) // 親要素（fieldset, section など）
  .getByRole('button', { name: 'ログイン' })
  .click();
```

### この書き方のメリット

- UI の構造に沿っているため読みやすい  
- テストケースが「ユーザーの操作手順」に近くなる  
- セレクタの衝突を防げる（同名ボタンが複数ある場合など）  
- CSS や DOM 構造の変更に強い

### 適用例

- セクション内のボタン  
- タブ内のフォーム  
- モーダル内の操作  
- カード内のアクション
